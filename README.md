# GADP — Governed Agentic Development Protocol
## Version 3.3

GADP is a protocol for building software with an AI coding tool that keeps the project governed, resumable, and production-ready across every session.

You describe what you are building. GADP figures out what it needs to be, designs it, scaffolds it, and builds it — one contract at a time, with a full audit trail, without losing context when a session ends.

---

## What this means in practice

You open your coding tool, say "hi", and pick up exactly where you left off. GADP knows whether you are still defining requirements, halfway through setting up the project, or three sprints into building it. You never paste a prompt. You never re-explain the project. You never wonder what the AI is about to change.

Every piece of functionality the project needs is written as a contract — a precise, testable definition of done. Code only ships when tests pass. Tests only pass when the contract says they should. The project cannot drift into something you did not agree to.

---

## How it works

GADP uses one entry point — the **Governor** — and six specialised agents that the Governor runs.

You talk to the Governor. The Governor reads the project's state, decides what needs to happen, runs the right agent, and tells you what was done in plain language. You never interact with agents directly.

**The six agents fall into two groups:**

**Setup agents** — run once per project, executed inline by the Governor as a natural conversation:

- **Intent Architect** — understands what you are building and produces the complete intent store: capability intents, quality targets, design tokens, screens, and journey
- **Outcome Resolver** — turns intents into executable contracts: architecture decisions, threat model, outcome contracts, invariants, and OpenAPI spec
- **Project Setup** — scaffolds the project: framework initialisation, dependencies, environment config, test stubs, CI/CD pipeline, and Sprint 0 verification

**Development agents** — run repeatedly across sprints, dispatched as isolated sub-agents:

- **Builder** — implements contracts one at a time, runs tests, fixes failures, and marks contracts passing
- **Auditor** — checks invariants, catches regressions, owns the audit log, and gates sprint transitions
- **Planner** — handles new features, architecture changes, sprint planning, and contract revisions

Every session starts the same way: the Governor reads `RESUME.md`, works out where the project is, and either picks up in progress or asks what you want to do. The project's state lives in files on disk — not in the conversation.

## Setting up a new project

**1. Clone the GADP repository into your project folder**

```bash
git clone https://github.com/syncandsolve/gadp-product-builder.git my-project
cd my-project
```

This gives you everything GADP needs — `AGENTS.md`, all six agent files, config, scripts, and skills. Everything else (`RESUME.md`, `intents/`, `outcomes/`, `decisions/`, `tests/`, `src/`) is generated by the Governor and agents as you work.

**2. Reinitialise git for your own project**

```bash
rm -rf .git
git init
git add .
git commit -m "init: add GADP protocol"
```

**3. (Optional) Generate initial project files from a config**

If you want to pre-populate the GADP YAML files before the Governor starts, create a `project-init.json` and run:

```bash
python3 gadp/scripts/gadp_init_project.py --config project-init.json
```

This generates all nine GADP files (`intent-store.yaml`, `contracts.yaml`, `decisions.yaml`, etc.) from structured input and validates them. See the script's docstring for the full `project-init.json` schema.

You can also reuse a `project-init.json` from a previous project as a starting point — edit it, run `gadp_init_project.py`, and the new project has a clean scaffold.

**4. Open the project in your AI coding tool**

Claude Code, opencode, or any tool that reads `AGENTS.md`.

**5. Say anything to start**

```
hi
```

The Governor detects that no `RESUME.md` exists, creates one, and asks you to describe what you are building. From that point on, GADP runs the full setup process automatically.

## Resuming a session

Say anything:

```
hi
```

```
resume
```

The Governor reads `RESUME.md` and tells you what is in progress and what is next. If a phase was left mid-way, it asks if you want to continue. If you are in active development, it gives you a brief status and waits for direction.

---

## Session boundaries

GADP enforces three hard stops where you must start a new session before continuing. These exist to prevent context pressure from degrading output quality.

**Hard Stop 1 — Before Sprint 0 verification**
After project setup completes (all ten tasks done), the Governor tells you to start a new session before running Sprint 0 verification. Setup and verification must not share a session.

**Hard Stop 2 — Before Sprint 1 implementation**
After Sprint 0 verification passes, the Planner produces a sprint plan and you approve it. The Governor then tells you to start a new session before any Builder dispatch. Verification and implementation must not share a session.

**Hard Stop 3 — Builder context pressure**
After three contracts pass in a single session, the Governor recommends (and eventually requires) starting a new session for the remaining contracts.

These stops are enforced by the Governor reading `RESUME.md` state — they are not advisory notes. The Governor will not dispatch the next agent across a session boundary.

## What a session looks like

**Before the project is defined:**

> *"What are you building? Tell me the problem it solves, who it's for, and what makes it different — two to four sentences is plenty."*

The Intent Architect works through your product in batches, confirming each before moving on. When it is done, you have a complete intent store in `./intents/`.

**During architecture and contracts:**

> *"Here's the stack I'd recommend — Next.js, Prisma with PostgreSQL, JWT auth in httpOnly cookies. Here's why each choice makes sense and what I ruled out."*

You review the architecture decisions and approve with `/approve-decisions`. The Outcome Resolver generates contracts, runs STRIDE threat analysis, writes invariants, and produces an OpenAPI spec.

**During setup:**

> *"Everything looks good — 23 contracts, 8 in Sprint 1. What's your deployment target?"*

Project Setup scaffolds the framework, installs dependencies, wires CI/CD, and generates failing test stubs. Sprint 0 verification confirms everything boots cleanly.

**During development:**

> *"Seven contracts are passing — auth and user profiles are solid. One is blocked: password reset is waiting on a SendGrid API key in your .env. Up next is the dashboard screen."*

The Builder implements contracts one at a time. When a test fails it diagnoses before retrying. If you quit mid-session, the Governor reads `./tmp/builder-progress.yaml` on the next session to determine exactly where to resume — down to the individual sub-task.

---

## The approval gates

**`/approve-decisions`** — required before any architecture decision is locked, any decision is changed, or any contract's definition is revised. The Governor shows you exactly what will change and what the downstream effects are.

**`/approve-sprint-[N]`** — required before any sprint's implementation begins. The Planner proposes the sprint, grouped by theme, and you confirm it.

**`/approve-deploy-prod`** — required before any production deployment. The Governor verifies that every contract is passing, the audit is clean, every threat is mitigated, runbooks are populated, and monitoring is live. It will not accept this command until all conditions are met.

## The contract system

Every feature is a contract. A contract has three parts:

- **When** — what triggers it (an HTTP request, a user action, a load condition)
- **Given** — what must be true before it is triggered
- **Then** — exactly what must happen, written precisely enough for a test to verify without human interpretation

A contract is not passing until its test passes. Vague contracts produce vague tests produce vague software.

**Example contract:**

```
When:  POST /api/v1/auth/register
Given: Email is not already registered, password meets minimum requirements
Then:
  - HTTP 201
  - Body contains { user: { id: uuid, email: string, created_at: iso8601 } }
  - Set-Cookie header present with httpOnly flag
  - No Authorization header in response
  - Row inserted in users table with password_hash (not plaintext)
```

## The invariant system

Invariants are the rules that never bend. Derived from architecture decisions and security requirements, enforced on every commit through CI.

Violations come in two levels:

- **hard_stop** — blocks all work immediately. No new contracts begin. The sprint cannot advance.
- **audit_flag** — recorded and tracked. Work continues. The Planner creates a remediation contract.

The canonical invariants every UI project gets: no hardcoded hex values (Tailwind tokens only), no auth tokens in localStorage (httpOnly cookies only), no raw SQL (ORM only), no render-blocking scripts, all images have explicit dimensions, all interactive elements have accessible names, bundle size within target.

These cannot be disabled without `/approve-decisions`.

## The mutation scripts

All GADP YAML file changes go through the scripts in `./gadp/scripts/` — never direct YAML writes. The scripts validate schema and write atomically. All agent invocations reference them directly at `gadp/scripts/gadp_*.py` — no copying to a project-level directory is needed or done.

| Script | What it does |
|---|---|
| `gadp_init_project.py` | Generates all initial GADP YAML files for a new project from a config file |
| `gadp_update_contract.py` | Updates mutable fields on a single contract (status, blocked_on, implemented_at) |
| `gadp_append_contract.py` | Appends a new OC-* contract with full schema validation |
| `gadp_append_intent.py` | Appends a new CI-*, SI-*, or QI-* intent with type-specific validation |
| `gadp_update_intent_status.py` | Updates the status field of a single intent |
| `gadp_append_audit.py` | Appends a typed event to the audit log (append-only) |
| `gadp_validate.py` | Validates all GADP YAML files and runs cross-file consistency checks |

## Resuming mid-phase

If a session ends in the middle of a setup phase (e.g. capability intents confirmed but design tokens not yet confirmed), the next session picks up exactly where it left off.

This works because every user confirmation is written to `RESUME.md phase_progress.confirmed_data` before continuing. The `confirmed_data` block also stores `derived_context` — the model's reasoning at each step — so a resuming session does not need to re-derive things like regulatory classification, direction selection rationale, or stack choice reasoning from scratch.

If a Builder session ends mid-contract, `./tmp/builder-progress.yaml` records every atomic sub-task completed (schema migration, repository layer, service layer, API handler, test run) and the test result at the time of interruption. The Governor reads this before the next Builder dispatch and handles the mid-flight state explicitly.

## Sprints

Sprint 1 ends only when a real person can complete the primary user journey without encountering a blank screen, placeholder page, or raw error message. "Code that compiles" is not done. "Walkable by a person" is done.

Before Sprint 2 begins, the Auditor runs the sprint gate: every contract passing, every invariant clean, performance baseline taken, First Run Standard verified. If anything fails, Sprint 2 does not begin.

Each sprint plan is proposed by the Planner, grouped by theme, and approved by you before any implementation starts.

## Regulatory compliance

GADP detects regulatory exposure automatically and adds the relevant quality intents and compliance contracts.

- **GDPR** — data deletion contracts, finite retention periods, consent management
- **HIPAA** — ePHI encryption at rest and in transit, immutable access audit log
- **PCI-DSS** — cardholder data never touches application servers, TLS 1.2+ enforced
- **SOC 2** — change management in audit log, quarterly access reviews

These are Sprint 1 scope when the exposure exists.

## Compatible tools

GADP works with any AI coding tool that reads an `AGENTS.md` file in the project root:

- Claude Code (recommended)
- opencode
- Any tool following the `AGENTS.md` convention

No specific model is required. GADP does not route by model name.

**Subagent dispatch requirement:** The Builder, Auditor, and Planner are dispatched as isolated sub-agents. This requires your coding tool to support a subagent Task tool (e.g. Claude Code's `Task` tool) or equivalent process-spawning mechanism. Without this, the Governor will write the dispatch prompt to `./tmp/dispatch-[agent]-[timestamp].md` and ask you to start a new agent session manually with that file.

## Skills

GADP includes implementation skill guides in `./gadp/skills/`. Each skill is a focused, opinionated guide for a specific implementation concern. Skills are read by sub-agents during execution — not by the Governor.

| Skill | Used by | When |
|---|---|---|
| `frontend-design` | Builder | Any UI contract (full_stack_pair set or SCREEN-* reference) |

Skills are versioned with the GADP repository and used directly — no copying required.

## The GADP guarantee

At any point in development, a fresh session with no conversation history can open this project and know:

- Exactly what the project is building and why
- What has been decided and what those decisions ruled out
- What is passing, what is blocked, and what is next
- Which threats have been mitigated and which compliance obligations remain
- What the primary user journey is and whether it is currently walkable

That is what `RESUME.md`, `intent-store.yaml`, `contracts.yaml`, `decisions.yaml`, `threat-model.yaml`, and `invariants.yaml` exist to provide. The project is self-describing. The session is always resumable. The work is always governed.
