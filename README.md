# GADP — Governed Agentic Development Protocol
## Version 3.0

GADP is a protocol for building software with an AI coding tool that keeps the project governed, resumable, and production-ready across every session.

You describe what you are building. GADP figures out what it needs to be, designs it, scaffolds it, and builds it — one contract at a time, with a full audit trail, without losing context when a session ends.

---

## What this means in practice

You open your coding tool, say "hi", and pick up exactly where you left off. GADP knows whether you are still defining requirements, halfway through setting up the project, or three sprints into building it. You never paste a prompt. You never re-explain the project. You never wonder what the AI is about to change.

Every piece of functionality the project needs is written as a contract — a precise, testable definition of done. Code only ships when tests pass. Tests only pass when the contract says they should. The project cannot drift into something you did not agree to.

---

## How it works

GADP uses one entry point — the **Governor** — and six specialised agents that the Governor dispatches.

You talk to the Governor. The Governor reads the project's state, decides what needs to happen, dispatches the right agent to do it, and tells you what was done in plain language. You never interact with agents directly.

**The six agents:**

- **Intent Architect** — understands what you are building and produces the complete intent store: capability intents, quality targets, design tokens, screens, and journey
- **Outcome Resolver** — turns intents into executable contracts: architecture decisions, threat model, outcome contracts, invariants, and OpenAPI spec
- **Project Setup** — scaffolds the project: framework initialisation, dependencies, environment config, test stubs, CI/CD pipeline, and Sprint 0 verification
- **Builder** — implements contracts one at a time, runs tests, fixes failures, and marks contracts passing
- **Auditor** — checks invariants, catches regressions, owns the audit log, and gates sprint transitions
- **Planner** — handles new features, architecture changes, sprint planning, and contract revisions

Every session starts the same way: the Governor reads `RESUME.md`, works out where the project is, and either picks up in progress or asks what you want to do. The project's state lives in files on disk — not in the conversation.

## Setting up a new project

**1. Create your project folder and initialise git**

```bash
mkdir my-project
cd my-project
git init
```

**2. Copy the GADP repository files into your project**

Create this folder structure manually:

```
my-project/
├── AGENTS.md                         ← copy from GADP repo root
└── gadp/
    ├── agents/
    │   ├── intent-architect.md       ← copy from gadp/agents/
    │   ├── outcome-resolver.md
    │   ├── project-setup.md
    │   ├── builder.md
    │   ├── auditor.md
    │   └── planner.md
    └── config/
        ├── framework-globs.yaml      ← copy from gadp/config/
        ├── qi-mandatory.yaml
        └── invariant-defaults.yaml
```

That is all you copy. Everything else — `RESUME.md`, `intents/`, `outcomes/`, `decisions/`, `scripts/`, `tests/`, `src/` — is generated by the Governor and agents as you work.

**3. Open the project in your AI coding tool**

Claude Code, opencode, or any tool that reads `AGENTS.md`.

**4. Say anything to start**

```
hi
```

```
let's start
```

```
I want to build a tool that does X
```

The Governor detects that no `RESUME.md` exists, creates one, and asks you to describe what you are building. From that point on, GADP runs the full setup process — no prompts to paste, no modes to switch.

## Resuming a session

Say anything:

```
hi
```

```
resume
```

```
where were we
```

The Governor reads `RESUME.md` and tells you what is in progress and what is next. If a phase was left mid-way, it asks if you want to continue. If you are in active development, it gives you a brief status and waits for direction.

## What a session looks like

**Before the project is defined:**

> *"What are you building? Tell me the problem it solves, who it's for, and what makes it different — two to four sentences is plenty."*

After you describe it, the Intent Architect works through your product: the strategic position, who it is for, what it must do, what good looks like, and how it should feel. It presents things in batches and asks for confirmation before moving on. When it is done, you have a complete intent store in `./intents/`.

**During architecture and contracts:**

> *"Here's the stack I'd recommend for what you're building: Next.js, Prisma with PostgreSQL, JWT auth in httpOnly cookies, deployed to Vercel. Here's why each choice makes sense for your goals — and what I ruled out."*

You approve the decisions. The Outcome Resolver generates contracts for every capability, runs STRIDE threat analysis, writes invariants, and produces an OpenAPI spec.

**During setup:**

> *"Everything looks good. You're building a web SaaS — 23 contracts, 8 Sprint 1, 12 Sprint 2. What's your deployment target?"*

Project Setup scaffolds the framework, installs dependencies, wires CI/CD, generates test stubs, and verifies the project can boot. By the end, every contract has a failing test stub waiting for implementation.

**During development:**

> *"Seven contracts are passing — auth and user profiles are solid. One is blocked: password reset is waiting on a SendGrid API key in your .env. Up next is the dashboard screen."*

The Builder implements contracts one at a time. When a test fails it diagnoses before retrying. When something genuinely cannot proceed it tells you exactly why and what is needed. The Auditor runs invariant checks after every few contracts and gates sprint transitions.

## The approval gates

Two things require your explicit approval:

**`/approve-decisions`** — required before any architecture decision is locked, any decision is changed, or any contract's definition is revised. The Governor shows you exactly what will change and what the downstream effects are. You say `/approve-decisions` to confirm.

**`/approve-deploy-prod`** — required before any production deployment. The Governor verifies that every contract is passing, the audit is clean, every threat is mitigated, runbooks are populated, and monitoring is live. It will not accept this command until all conditions are met.

## The contract system

Every feature is a contract. A contract has three parts:

- **When** — what triggers it (an HTTP request, a user action, a load condition)
- **Given** — what must be true before it is triggered
- **Then** — exactly what must happen, written precisely enough for a test to verify without human interpretation

A contract is not passing until its test passes. A test is not meaningful until its assertions are specific. Vague contracts produce vague tests produce vague software.

**Example of a contract done right:**

```
When:  POST /api/v1/auth/register
Given: Email is not already registered, password meets minimum requirements
Then:
  - HTTP 201
  - Body contains { user: { id: uuid, email: string, created_at: iso8601 } }
  - Set-Cookie header present with httpOnly flag
  - No Authorization header in response
  - Row inserted in users table with password_hash (not plaintext)
```

Contracts are created by the Outcome Resolver before any code is written. The Builder implements them in sprint order. The Auditor checks that passing contracts stay passing. The Planner revises them when requirements change.

## The invariant system

Invariants are the rules that never bend. They are derived from architecture decisions and security requirements and enforced on every commit through CI.

Each invariant either runs a detection command automatically or flags itself for manual review. Violations come in two levels:

- **hard_stop** — blocks all work immediately. No new contracts begin until this is cleared. The sprint cannot advance.
- **audit_flag** — recorded and tracked. Work continues. The Planner creates a remediation contract.

The canonical invariants every UI project gets:

- No hardcoded hex values in source — Tailwind theme tokens only
- No auth tokens in localStorage — httpOnly cookies only
- No raw SQL — ORM only
- No render-blocking scripts
- All images have explicit dimensions
- All interactive elements have accessible names
- Bundle size within target

These cannot be disabled without `/approve-decisions`. They exist because the cost of violating them is always higher than the cost of following them.

## Sprints

GADP organises work into sprints. Sprint 1 is the most important — it ends only when a real person can complete the primary user journey without encountering a blank screen, placeholder page, or raw error message. Sprint 1 defines "working software", not "code that compiles."

Before Sprint 2 begins, the Auditor runs the sprint gate: every contract passing, every invariant clean, performance baseline taken, First Run Standard verified. If anything fails, Sprint 2 does not begin.

Each sprint plan is proposed by the Planner and approved by you before any implementation starts. The plan groups contracts by theme and tells you in one sentence what the sprint delivers. You approve with `/approve-sprint-[N]`.

## Supported product types

GADP adapts its setup, contracts, and invariants to the product type it detects from your description.

Web SaaS · Marketing site · Chrome extension · Desktop app (Electron, Tauri) · CLI tool · API product · Internal tool · Mobile-first PWA · Composite (combinations of the above)

Each type gets the right stack recommendations, the right mandatory quality intents, the right CI/CD configuration, and the right Sprint 1 definition.

---

## Regulatory compliance

GADP detects regulatory exposure from your project description and adds the relevant quality intents and compliance contracts automatically.

- **GDPR** — data deletion contracts, finite retention periods, consent management
- **HIPAA** — ePHI encryption at rest and in transit, immutable access audit log, minimum necessary access controls
- **PCI-DSS** — cardholder data never touches application servers, TLS 1.2+ enforced
- **SOC 2** — change management in audit log, quarterly access reviews

These are not optional extras — they are part of Sprint 1 scope when the exposure exists.

## Compatible tools

GADP works with any AI coding tool that reads an `AGENTS.md` file in the project root:

- Claude Code (recommended)
- opencode
- Any tool following the `AGENTS.md` convention

No specific model is required. GADP does not route by model name — you configure the model in your tool.

## The GADP guarantee

At any point in development, a fresh session with no conversation history can open this project and know:

- Exactly what the project is building and why
- What has been decided and what those decisions ruled out
- What is passing, what is blocked, and what is next
- Which threats have been mitigated and which compliance obligations remain
- What the primary user journey is and whether it is currently walkable

That is what `RESUME.md`, `intent-store.yaml`, `contracts.yaml`, `decisions.yaml`, `threat-model.yaml`, and `invariants.yaml` exist to provide. The project is self-describing. The session is always resumable. The work is always governed.
